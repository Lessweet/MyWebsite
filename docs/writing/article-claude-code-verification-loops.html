验证循环
把手动检查写进 Skill

Agent 协作
2026-07-23阅读约 6 分钟

用 coding Agent 生成设计的这个过程中，基本每次生成后，都要“截图+修改prompt”这个手动检验环节。

大多数 agentic coding session 都遵循一个循环： agent 收集上下文，采取行动，验证结果，并在需要时回到循环中收集更多上下文。而 agent 无法推断的内容成为了手动检查功能的步骤，这些手动步骤可以转化为验证循环。

Anthropic 把这个过程叫验证循环（verification loop）——每次重复做的那些检查步骤，可以编成 Skill 让 Claude 自己执行。

验证 Skill
把检查习惯写成规矩

Agentic coding 的循环是这样的：发出请求 → 收集上下文 → 执行动作 → 验证结果 → 不对就再来一轮。Claude Code 内置了一批验证能力——从 /verify Skill 到 Toolchain 错误捕获到 PR 级的 Code Review——但这些查的是通用规范。每次手动重复做的、跟项目绑定的检查，它不知道。

博客里给的思路是：把每次实现新功能之后重复执行的手动检查记下来，用大白话写出来，就像第一天交给新队友的那份文档。 不光是定性的判断能写，确定性的死规则一样好使——「拒绝任何没有 backfill 就删除列的数据库迁移」，这种 linter 管不到但项目里必须遵守的规矩，写成 Skill 反而更合适。

博客给了一个最小验证 Skill 的例子，检查日志规范的：

# .claude/skills/verify-log-hygiene/SKILL.md
---
name: verify-log-hygiene
description: 检查错误日志包含请求 ID 且从不包含请求体。
allowed-tools: [Read, Edit, Grep]
---
读取当前 diff 中的错误处理路径。
对每个错误路径上的日志调用，确认它包含请求 ID，
且不传递请求体、headers 或任何用户提供的 payload。
按 file:line 报告每处违规，然后修复。

就是一段白话 markdown，没有代码逻辑、没有正则。Claude 读到就知道该查什么、怎么查、查到问题怎么处理。你脑子里那套「这里应该怎样」的标准，写下来就行，不用翻译成代码。

四种挂法
从个人习惯到团队标准

验证 Skill 写好了，怎么挂到工作流里？博客给了四种方式，从轻到重。

•独立调用（Standalone），想起来手动跑一下。适合安全扫描、无障碍审计这类横切检查，不需要每次改完都跑。缺点是得自己记着调；如果发现每次都要手动跑一遍，说明该往下挪了。

•嵌入式（Embedded），在生成型 Skill 末尾追一句话。比如组件脚手架 Skill 的末尾加一行「创建组件文件后，对其运行 eslint，解决所有报错后再报告完成」——验证就自动成了生成流程的一部分。限制是只能改自己维护的 Skill，内置的改不了。

•链式（Chained），一个 Skill 跑完自动调下一个。博客提到 Anthropic 的 Claude Code 团队日常就在用这个模式：先跑 /code-review 找 bug，再跑 /simplify 清理 diff，然后 /verify 确认功能，最后走自定义的 /design Skill 检查 UI 规范——四步串成一条链，习惯变成了合约。但 token 消耗会明显增加，博客建议广泛部署前先测几轮。

•每个 PR 上，链式验证稳定之后推到团队级——每个 PR 自动过同一套标准，不管谁提交都走同样的关卡。博客的建议是别急着推：验证流程没稳定就上到 PR 级，每次调整都是团队可见事件。

有个小细节能说明问题——Anthropic 团队链式流程里那步 /design Skill，查的是 UI 规范。设计标准同样能编成验证 Skill：比如组件命名有没有对齐设计系统、间距有没有跟规范走、配色有没有跑出色板——review 时人眼扫的那些检查，同样可以写成一条自动验证。

Skill 越写越多
维护也是成本

编进 Skill 的检查越多，Claude 第一次交出来的结果越接近标准——因为它知道最后要过哪些关，执行时就会主动对齐。

但 Skill 不是写完就完事的。链式验证跑一轮，token 消耗比单个 Skill 高出不少；Skill 本身也需要迭代，第一版写出来大概率要在新任务上试几轮、发现遗漏、再改。博客给了个具体的起步建议：挑出这周重复次数最多的那一个手动收尾步骤，先试内置的 /verify Skill，不够用再用大白话写一条自定义的，在新任务上跑一遍确认效果。

【补充亲历：自己写验证 Skill 的一个实际场景——比如给写作流程、图标检查加过什么验证，第一版效果怎样、迭代了几轮】

高频且固定的手动检查——每次都会犯的同类错误、每次都要过的格式规范——编成验证 Skill 省掉的时间实打实，值得放进日常。但标准还在变的检查、或者偶尔才做一次的审查，维护成本可能比手动跑还高，不急着编。过期的 Skill 跑出来的结果比不跑更麻烦——以为检查过了，其实查的是过时的规矩。

Claude Code 团队的 Delba de Oliviera 在博客结尾写了一段话：那些不再需要你亲手修的部分，把你的注意力释放出来，留给了只有你能做的、没有任何 Skill 能替代的工作。
